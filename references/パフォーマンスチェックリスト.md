## Core Web Vitals の目標値

| 指標 | 良好 | 要改善 | 不良 |
|--------|------|------------|------|
| LCP (Largest Contentful Paint) | ≤ 2.5s | ≤ 4.0s | > 4.0s |
| INP (Interaction to Next Paint) | ≤ 200ms | ≤ 500ms | > 500ms |
| CLS (Cumulative Layout Shift) | ≤ 0.1 | ≤ 0.25 | > 0.25 |

## TTFB の診断

TTFB が遅い（> 800ms）ときは、DevTools の Network waterfall で各要素を確認します。

- [ ] **DNS 解決** が遅い → 既知のオリジンに `<link rel="dns-prefetch">` または `<link rel="preconnect">` を追加する
- [ ] **TCP/TLS ハンドシェイク** が遅い → HTTP/2 を有効にし、edge 配備を検討し、keep-alive を確認する
- [ ] **サーバー処理** が遅い → バックエンドをプロファイルし、遅いクエリを確認し、キャッシュを追加する

## フロントエンド・チェックリスト

### 画像
- [ ] 画像にモダン形式（WebP, AVIF）を使っている
- [ ] 画像サイズがレスポンシブに最適化されている（`srcset` と `sizes`）
- [ ] 画像と `<source>` 要素に明示的な `width` と `height` がある（art direction 時の CLS を防ぐ）
- [ ] ファーストビュー外の画像に `loading="lazy"` と `decoding="async"` を使っている
- [ ] ヒーロー / LCP 画像に `fetchpriority="high"` を付け、lazy loading しない

### JavaScript
- [ ] バンドルサイズが gzip 後 200KB 未満（初回読み込み）
- [ ] ルートや重い機能に動的 `import()` でコード分割している
- [ ] Tree shaking が有効（依存が ESM を配布し、`sideEffects: false` を持つことを確認）
- [ ] `<head>` にブロッキング JavaScript がない（`defer` または `async` を使う）
- [ ] 重い計算を Web Worker に逃がしている（必要な場合）
- [ ] 同じ props で再レンダーされる高コストコンポーネントに `React.memo()` を使っている
- [ ] `useMemo()` / `useCallback()` は、プロファイリングで効果が確認できた箇所だけで使っている

### CSS
- [ ] クリティカル CSS をインライン化または preload している
- [ ] 非クリティカルなスタイルで render-blocking CSS がない
- [ ] 本番で CSS-in-JS のランタイムコストがない（抽出を使う）
- [ ] フォント表示戦略が設定されている（`font-display: swap` または `optional`）
- [ ] カスタムフォントの前に system font stack を検討している

### ネットワーク
- [ ] 静的アセットを長い `max-age` と content hash 付きでキャッシュしている
- [ ] 必要に応じて API レスポンスをキャッシュしている（`Cache-Control`）
- [ ] HTTP/2 または HTTP/3 が有効
- [ ] 既知のオリジンに `<link rel="preconnect">` を設定している
- [ ] 不要なリダイレクトがない

### レンダリング
- [ ] layout thrashing（強制同期レイアウト）がない
- [ ] アニメーションに `transform` と `opacity` を使っている（GPU 加速）
- [ ] 長いリストで virtualization（例: `react-window`）を使っている
- [ ] 不要な全画面再レンダーがない

## バックエンド・チェックリスト

### データベース
- [ ] N+1 クエリがない（eager loading / join を使う）
- [ ] クエリに適切なインデックスがある
- [ ] 一覧系エンドポイントがページネーションされている（`SELECT * FROM table` をしない）
- [ ] コネクションプーリングが設定されている
- [ ] 遅いクエリのログが有効

### API
- [ ] 応答時間が p95 で 200ms 未満
- [ ] リクエストハンドラ内で重い同期処理をしていない
- [ ] 個別呼び出しのループではなく一括操作を使っている
- [ ] レスポンス圧縮（gzip/brotli）が有効
- [ ] 適切なキャッシュがある（in-memory, Redis, CDN）

### インフラ
- [ ] 静的アセット用 CDN がある
- [ ] サーバーがユーザーに近い場所にある（または edge 配備）
- [ ] 必要なら水平スケーリングが設定されている
- [ ] ロードバランサー向けのヘルスチェックエンドポイントがある

## 計測コマンド

```bash
# Lighthouse CLI
npx lighthouse https://localhost:3000 --output json --output-path ./report.json

# バンドル解析
npx webpack-bundle-analyzer stats.json
# または Vite:
npx vite-bundle-visualizer

# バンドルサイズ確認
npx bundlesize

# コード内で Web Vitals を計測
import { onLCP, onINP, onCLS } from 'web-vitals';
onLCP(console.log);
onINP(console.log);
onCLS(console.log);
```

## よくあるアンチパターン

| アンチパターン | 影響 | 修正 |
|---|---|---|
| N+1 クエリ | DB 負荷が線形に増える | joins、includes、一括読み込みを使う |
| 無制限クエリ | メモリ枯渇、タイムアウト | 常にページネーションし、LIMIT を付ける |
| インデックス不足 | データ増加に伴い読み取りが遅くなる | 絞り込み / 並び替えカラムにインデックスを追加する |
| Layout thrashing | カクつき、フレーム落ち | DOM 読み取りをまとめ、その後に書き込みをまとめる |
| 最適化されていない画像 | LCP 悪化、帯域浪費 | WebP、レスポンシブサイズ、lazy load を使う |
| 大きすぎるバンドル | Time to Interactive が遅くなる | コード分割、tree shaking、依存棚卸しを行う |
| メインスレッドを塞ぐ処理 | INP 悪化、UI が反応しない | Web Worker を使い、作業を遅延させる |
| メモリリーク | メモリが増え続け最終的にクラッシュする | listener、interval、ref の後始末をする |
