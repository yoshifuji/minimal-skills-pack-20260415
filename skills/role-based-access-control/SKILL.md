# Role Based Access Control

## 概要

認証は「誰か」を決める機能で、認可は「何をしてよいか」を決める機能です。ここを曖昧にすると、見えてはいけないデータが見えたり、編集できてはいけないデータが更新されたりします。UI で隠すだけでは足りません。**最終判定は必ず server 側で行う**前提で組み立ててください。

## コアプロセス

1. **まず役割を定義する**
   - guest
   - user
   - manager
   - admin
   - project に必要な役割だけを持つ

2. **役割と操作の対応表を作る**
   - 誰が一覧を見られるか
   - 誰が詳細を見られるか
   - 誰が作成できるか
   - 誰が更新できるか
   - 誰が削除できるか

3. **所有者チェックを分けて考える**
   - role が合っているか
   - resource owner か
   - 管理者だけ例外か
   - 所有関係は DB で追えるようにする

4. **UI と server で同じルールを使う**
   - UI は見せ分けに使う
   - server は最終判定に使う
   - helper にして重複を減らす

5. **保護ページと API を同じ基準で守る**
   - page 入口で user を確認する
   - API でも同じ条件を確認する
   - route ごとに条件がずれないようにする

6. **forbidden と not found を意図的に分ける**
   - 見えてよい存在か
   - 存在しないのか
   - 存在はするが触れないのか
   - project の方針を先に決める

役割表は `../../references/権限表ひな形.md`、所有者確認は `../../references/所有者確認チェックリスト.md`、閲覧制限ページの考え方は `../../references/閲覧制限ページガイド.md` を参照してください。

## よくある正当化

| Rationalization | Reality |
|---|---|
| 「画面にボタンを出していないから安全」 | API を叩けるなら安全ではありません。 |
| 「admin だけ例外で全部通せばよい」 | 例外が多いほど境界が壊れます。必要最小限にしてください。 |
| 「owner check は画面側だけで十分」 | server 側で見ない owner check は存在しないのと同じです。 |
| 「role が 1 つなら認可は不要」 | 自分のデータだけ触れる、の時点で認可は必要です。 |

## レッドフラグ

- page と API で条件が違う
- owner check が UI にしかない
- admin 判定が string 比較で散らばっている
- forbidden と not found の扱いが endpoint ごとに違う
- role の定義が増えたのに対応表がない

## 検証

- [ ] 役割と操作の対応表がある
- [ ] owner check が server 側で行われている
- [ ] UI と API が同じ認可ルールを使っている
- [ ] 管理者例外の条件を説明できる
- [ ] page と route の保護条件が一致している
