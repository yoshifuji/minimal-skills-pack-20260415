# 汎用サービス(予約マーケットプレイス)仕様プロンプト

このファイル名は互換性のため維持しています。  
中身は、`docs/skills汎用化再構成方針書.md` に沿って再構成した **汎用サービス予約マーケットプレイス向け example overlay** です。

以下を **そのまま AI コーディングエージェントに渡してください**。  
目的は、単一 provider 向けの予約、Stripe Checkout、Stripe Connect Express、refund、webhook 整合を備えた日本語 UI のサービス予約マーケットプレイスを、仕様から実装することです。

## この pack で開発しやすいアプリ例

この pack は、次のような「provider が listing を公開し、customer が将来日時を指定して予約・決済する」構成に向いています。

| アプリ例 | provider | customer | listing | 適合理由 |
|---|---|---|---|---|
| `しっぽケア予約` | ペットシッター | 飼い主 | お世話プラン | 将来予約、単一 provider 決済、キャンセル返金と相性が良い |
| `くらしクリーン予約` | クリーナー | 依頼者 | 清掃プラン | role 別 portal、訪問日時予約、provider 受取に適合 |
| `伴走トレーニング予約` | トレーナー | 会員 | セッションプラン | 継続しやすい公開プロフィール型 marketplace |
| `相談デスク予約` | 相談員 | 相談者 | 相談セッション | オンライン相談でも同じ checkout / refund パターンを使える |
| `おうちシェフ便` | シェフ | 依頼者 | コースプラン | 将来日時予約、provider payout、refund に適合 |

このファイルでは、既定例として **ペットシッター予約マーケットプレイス** を実装対象にします。

## DB モデル

Supabase SQL で次を作成すること。  
テーブル名は、別アプリへ転用しやすいよう中立名を使う。

### 1. `profiles`

- `id uuid primary key` references `auth.users`
- `full_name text not null`
- `role text not null default 'customer'`
- role の allowed values:
  - `customer`
  - `provider`
  - `admin`
- `created_at`, `updated_at`

### 2. `provider_profiles`

- `id uuid primary key` references `profiles`
- `headline text not null default 'New provider'`
- `bio text not null default ''`
- `service_area text not null default ''`
- `supported_pet_types text[] not null default '{}'`
- `stripe_account_id text unique`
- `details_submitted boolean not null default false`
- `charges_enabled boolean not null default false`
- `payouts_enabled boolean not null default false`
- `last_stripe_sync_at timestamptz`
- `created_at`, `updated_at`

### 3. `service_listings`

- `id uuid primary key default gen_random_uuid()`
- `provider_id uuid not null`
- `title text not null`
- `description text not null default ''`
- `duration_minutes integer not null` with check `> 0 and <= 480`
- `price_amount integer not null` with check `> 0`
- `currency text not null default 'jpy'`
- `pet_type text not null default 'dog'`
- `is_active boolean not null default true`
- `created_at`, `updated_at`

### 4. `reservations`

- `id uuid primary key default gen_random_uuid()`
- `service_listing_id uuid not null`
- `provider_id uuid not null`
- `customer_id uuid not null`
- `scheduled_for timestamptz not null`
- `pet_name text not null default ''`
- `pet_type text not null default ''`
- `notes text not null default ''`
- `status text not null default 'pending'`
- allowed reservation statuses:
  - `pending`
  - `paid`
  - `expired`
  - `cancelled`
  - `payment_failed`
- `price_amount integer not null`
- `currency text not null default 'jpy'`
- `application_fee_amount integer not null default 0`
- `stripe_checkout_session_id text unique`
- `stripe_payment_intent_id text`
- `stripe_payment_status text`
- `cancelled_at timestamptz`
- `stripe_refund_id text unique`
- `refund_status text`
- `refund_amount integer`
- `refunded_at timestamptz`
- `refund_failure_reason text`
- `created_at`, `updated_at`

### 5. `public_provider_directory`

重要:

- SECURITY DEFINER view ではなく、**公開列だけを持つ table** にする
- `profiles` と `provider_profiles` の変更を trigger で同期する
- 公開可能な列だけを持たせる
- 少なくとも以下の列:
  - `id`
  - `full_name`
  - `headline`
  - `bio`
  - `service_area`
  - `supported_pet_types`
  - `details_submitted`
  - `charges_enabled`
  - `payouts_enabled`

### 6. 共通 DB 要件

- `updated_at` 自動更新 trigger を用意する
- `auth.users` insert 時に `profiles` を自動作成する trigger を用意する
- `role = provider` の場合は `provider_profiles` も自動作成する
- 適切な index を付ける
  - `stripe_account_id`
  - `provider_id`
  - `customer_id`
  - `stripe_checkout_session_id`
  - `stripe_refund_id`

## RLS ポリシー

必須:

- `public_provider_directory` は `anon`, `authenticated` に公開 read
- `profiles`
  - user は own profile read/update 可
- `provider_profiles`
  - provider は own row read/insert/update 可
- `service_listings`
  - public は `is_active = true` のもの read 可

- `reservations`

## Stripe Connect 仕様

### Connect account

- connected account type: `express`
- country は環境変数から
- `business_type: "individual"`
- request capabilities:
  - `card_payments`
  - `transfers`
- provider ごとに account を 1 つ持つ
- 初回作成後に `provider_profiles.stripe_account_id` に保存する

### Onboarding

- `accountLinks.create()`
- `type: "account_onboarding"`
- `refresh_url` はアプリ側 refresh ルート
- `return_url` は provider ダッシュボード

### 状態同期

- `Stripe.Account` から次を DB へ同期:
  - `details_submitted`
  - `charges_enabled`
  - `payouts_enabled`
  - `last_stripe_sync_at`
- `account.updated` webhook でも同期する
- webhook 未設定でも、provider ダッシュボードの「状態同期」で手動回復できるようにする

### Express Dashboard

- 接続済み account に対して `accounts.createLoginLink()`

## 予約・決済仕様

### 予約前提

- 予約可能なのは `customer` role のみ
- provider の `charges_enabled` と `payouts_enabled` が true でなければ予約不可
- listing は `is_active = true` のみ予約可

### Checkout Session 作成

フロー:

1. customer 認証を確認
2. service listing を取得
3. provider の connected account 有無を確認
4. Stripe account を最新同期
5. readiness 不足なら予約拒否
6. reservation を `pending` で先に作成
7. `application_fee_amount` を `PLATFORM_FEE_BPS` から計算
8. Checkout Session を作成
9. `stripe_checkout_session_id` を reservation に保存
10. session URL を返す

### Checkout 条件

- `mode: "payment"`
- 1 reservation = 1 line item
- metadata に少なくとも次を入れる:
  - `reservationId`
  - `listingId`
  - `providerId`
  - `customerId`
- `payment_intent_data` では:
  - `application_fee_amount`
  - `transfer_data.destination`
  - `on_behalf_of`
  - metadata
- `success_url`:
  - `/checkout/success?session_id={CHECKOUT_SESSION_ID}`
- `cancel_url`:
  - `/checkout/cancel?reservation_id={reservation.id}`

### 失敗時の扱い

- reservation 作成後に Checkout 作成で失敗した場合は、その reservation を `cancelled` に寄せてよい
- internal error で予約行だけ残り続けないようにする

## Checkout 結果ページ

### success

- `session_id` があれば、best effort で session を取得して reservation を `paid` に寄せる
- ただし Stripe を source of truth とし、webhook が本命
- webhook 未到達時の UX recovery として使う

### cancel

- `reservation_id` を受け取る
- ログイン中の customer 本人の reservation で、かつ `pending` のときだけ `cancelled` に更新
- すでに `paid` / `expired` / `cancelled` 等なら状態は変えない
- ログインしていない場合は DB 更新を強制しない

## Stripe Webhook 仕様

### runtime

- Node.js runtime を明示する
- raw body で署名検証する

### 署名検証

- `stripe-signature` header 必須
- `STRIPE_WEBHOOK_SECRET` で `constructEvent()`

### 受けるイベント

必須:

- `checkout.session.completed`
- `checkout.session.async_payment_succeeded`
- `checkout.session.async_payment_failed`
- `checkout.session.expired`
- `refund.created`
- `refund.updated`
- `refund.failed`
- `account.updated`

### reservations 更新方針

`checkout.session.*` の処理では次の設計を入れること。

- まず `metadata.reservationId` で更新を試みる
- 見つからなければ `stripe_checkout_session_id = session.id` にフォールバックする

状態更新ルール:

- completed / async_payment_succeeded → `paid`
- expired → `expired`
- async_payment_failed → `payment_failed`

保存する値:

- `stripe_payment_status`
- `stripe_payment_intent_id`（支払い成功系）
- 可能なら `status` に応じた値

### refund webhook 更新方針

- まず refund metadata の `reservationId` で reservation を探す
- なければ `stripe_refund_id = refund.id` で更新を試みる

refund 系で保存する値:

- `stripe_refund_id`
- `refund_status`
- `refund_amount`
- `refund_failure_reason`
- `refunded_at`
- refund が active なら reservation `status` は `cancelled`
- refund が failed / canceled なら reservation `status` は `paid` 側へ戻す

### 重要な運用方針

- DB 更新に失敗したら webhook で **500 を返す**
- 理由: Stripe の再送を使って整合性回復したいから

## customer キャンセルと返金仕様

### cancel 可能条件

- ログイン中 user が customer
- reservation の `customer_id` が current user と一致
- reservation status が `paid`
- `scheduled_for` が現在時刻より未来
- `stripe_payment_intent_id` がある

### 実行内容

- `POST /api/reservations/[id]/cancel`
- Stripe の PaymentIntent を取得し latest charge を特定
- Refund を作成:
  - `charge`
  - `reverse_transfer: true`
  - `refund_application_fee: true`
  - `reason: "requested_by_customer"`
  - reservation / paymentIntent / customer / provider metadata を付与
- idempotency key を付ける
  - 例: `reservation-cancel-refund:{reservation.id}`
- refund 作成直後の payload から reservation を更新してよい
- 以後は refund webhook でも最新状態に揃える

### 返金 UI

- customer dashboard 一覧に `refund_status` を表示
- `pending`, `requires_action`, `succeeded`, `failed`, `canceled` を plain Japanese で表示する
