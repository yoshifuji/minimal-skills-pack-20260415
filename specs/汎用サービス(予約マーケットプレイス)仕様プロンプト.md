## 作るアプリ

### プロダクト名

- 画面表示名: `しっぽケア予約`
- 内容: ペットシッターがプロフィールとお世話プランを公開し、飼い主が日時を選んで予約し、Stripe Checkout で支払えるサービス予約アプリ

### 画面上の呼び方

- `provider` = `ペットシッター`
- `customer` = `飼い主`
- `service_plan` = `お世話プラン`
- `reservation` = `予約`

### スコープの考え方

- このアプリは「事業者と利用者の両方がいる予約サービス」として作る
- 支払いはプラットフォームが受け付ける
- 事業者への精算処理や自動送金は **今回は対象外**

## 技術スタック

必須:

- Next.js App Router
- React
- TypeScript
- Supabase Auth
- Supabase Postgres
- Supabase Storage
- `@supabase/ssr`
- Stripe Checkout
- Zod

方針:

- UI はプレーンな CSS を使う
- 状態管理は React 標準機能中心にする
- ORM は追加しない
- DB への読み書きは `lib/` 配下の共通処理へ寄せる
- route handler に業務ロジックをベタ書きしすぎない

## UI と文言の方針

- 日本語 UI
- 明るい背景
- 清潔感と安心感を優先する
- 過度に派手にしない
- 予約、支払い、返金の状態を利用者が理解しやすい言葉で出す
- 技術用語をそのまま出しすぎない
- 飼い主、ペットシッター、将来の運用担当が迷いにくい導線にする

## 主要ユーザー

- 飼い主
- ペットシッター
- 管理者

今回は管理者専用画面は不要です。  
ただし `admin` ロール自体は保持してください。

## 認証仕様

### サインアップ

- ルート: `/signup`
- 入力項目:
  - `fullName`
  - `email`
  - `password`
  - `role` (`customer` または `provider`)
- Supabase Auth のメールアドレスとパスワード認証を使う
- user metadata に `full_name` と `role` を保存する
- メール確認がある場合とない場合の両方に対応し、利用者が迷わない文言を出す

### ログイン

- ルート: `/login`
- email / password ログイン
- ログイン成功後の遷移:
  - `customer` → `/dashboard/customer`
  - `provider` → `/dashboard/provider`
  - `admin` → `/dashboard/provider` に仮遷移でもよいが、役割は保持する

### ログアウト

- ナビゲーションからログアウトできる

### セッション

- `@supabase/ssr` を使う
- ブラウザー用、サーバー用、管理用のクライアントを分ける
- `middleware.ts` で通常リクエスト時のセッション継続を扱う
- `/api/stripe/webhook` は署名検証のため、不要な処理を避ける

## DB モデル

テーブル名は将来の別アプリへ流用しやすい中立名を使ってください。

### 1. `profiles`

- `id uuid primary key` references `auth.users`
- `full_name text not null`
- `role text not null default 'customer'`
- role の許可値:
  - `customer`
  - `provider`
  - `admin`
- `created_at`
- `updated_at`

### 2. `provider_profiles`

- `user_id uuid primary key` references `profiles`
- `display_name text not null default ''`
- `introduction text not null default ''`
- `service_area text not null default ''`
- `avatar_image_path text`
- `is_public boolean not null default false`
- `accepting_new_reservations boolean not null default false`
- `created_at`
- `updated_at`

### 3. `service_plans`

- `id uuid primary key default gen_random_uuid()`
- `provider_id uuid not null`
- `title text not null`
- `description text not null default ''`
- `duration_minutes integer not null`
- `price_amount integer not null`
- `currency text not null default 'jpy'`
- `cover_image_path text`
- `is_active boolean not null default true`
- `created_at`
- `updated_at`

制約:

- `duration_minutes > 0 and duration_minutes <= 480`
- `price_amount > 0`

### 4. `reservations`

- `id uuid primary key default gen_random_uuid()`
- `service_plan_id uuid not null`
- `provider_id uuid not null`
- `customer_id uuid not null`
- `scheduled_for timestamptz not null`
- `pet_name text not null default ''`
- `pet_type text not null default ''`
- `notes text not null default ''`
- `status text not null default 'pending'`
- `price_amount integer not null`
- `currency text not null default 'jpy'`
- `stripe_checkout_session_id text unique`
- `stripe_payment_intent_id text`
- `stripe_payment_status text`
- `stripe_refund_id text unique`
- `refund_status text`
- `refund_amount integer`
- `cancelled_at timestamptz`
- `refunded_at timestamptz`
- `created_at`
- `updated_at`

status の許可値:

- `pending`
- `paid`
- `expired`
- `cancelled`
- `payment_failed`
- `refunded`

### 共通 DB 要件

- `updated_at` 自動更新 trigger を用意する
- `auth.users` insert 時に `profiles` を自動作成する
- role が `provider` の場合は `provider_profiles` も自動作成する
- 主要な検索列に index を付ける

最低限の index:

- `provider_profiles.is_public`
- `service_plans.provider_id`
- `service_plans.is_active`
- `reservations.provider_id`
- `reservations.customer_id`
- `reservations.scheduled_for`
- `reservations.stripe_checkout_session_id`
- `reservations.stripe_refund_id`

## 行レベルのアクセス制御

必須:

- `profiles`
  - 本人のみ read / update 可
- `provider_profiles`
  - 公開中 (`is_public = true`) の行は公開 read 可
  - provider 本人は own row を read / update 可
- `service_plans`
  - `is_active = true` かつ公開中 provider の plan は公開 read 可
  - provider 本人は own plans を作成、更新、停止できる
- `reservations`
  - customer 本人は own reservations を read / create 可
  - provider 本人は own incoming reservations を read 可

注意:

- クライアント側の表示制御だけで済ませない
- API 側でも role と所有者確認を行う

## 予約と決済

### 予約できる条件

- ログイン中 user が `customer`
- provider profile が公開中
- provider が予約受付中
- service plan が有効

### Checkout Session 作成

フロー:

1. customer 認証を確認する
2. service plan を取得する
3. provider の公開状態と受付状態を確認する
4. 価格と通貨をサーバー側で確定する
5. reservation を `pending` で先に作る
6. Stripe Checkout Session を作る
7. `stripe_checkout_session_id` を reservation に保存する
8. session URL を返す

条件:

- `mode: "payment"`
- 1 予約 = 1 line item
- 金額はクライアント入力を信用しない
- metadata には少なくとも次を入れる
  - `reservationId`
  - `servicePlanId`
  - `providerId`
  - `customerId`
- `success_url`:
  - `/checkout/success?session_id={CHECKOUT_SESSION_ID}`
- `cancel_url`:
  - `/checkout/cancel?reservation_id={reservation.id}`

### success ページ

- `session_id` があれば best effort で Stripe Session を確認し、reservation を `paid` に寄せてよい
- ただし本命は webhook とする

### cancel ページ

- `reservation_id` を受け取る
- ログイン中の customer 本人の reservation で、かつ `pending` のときだけ `cancelled` に更新する
- それ以外の状態は勝手に書き換えない

## Stripe Webhook

### 基本

- Node.js runtime を使う
- raw body で署名検証する
- `stripe-signature` header を必須にする
- `STRIPE_WEBHOOK_SECRET` で `constructEvent()` する

### 受けるイベント

- `checkout.session.completed`
- `checkout.session.async_payment_succeeded`
- `checkout.session.async_payment_failed`
- `checkout.session.expired`
- `refund.created`
- `refund.updated`
- `refund.failed`

### reservation 更新方針

- まず `metadata.reservationId` で予約を探す
- 見つからなければ `stripe_checkout_session_id = session.id` で探す

状態更新:

- `checkout.session.completed` → `paid`
- `checkout.session.async_payment_succeeded` → `paid`
- `checkout.session.async_payment_failed` → `payment_failed`
- `checkout.session.expired` → `expired`

保存する値:

- `stripe_payment_status`
- `stripe_payment_intent_id`
- `status`

### 返金 webhook 更新方針

- まず refund metadata の `reservationId` で探す
- 見つからなければ `stripe_refund_id = refund.id` で探す

保存する値:

- `stripe_refund_id`
- `refund_status`
- `refund_amount`
- `refunded_at`

状態更新:

- refund 完了時は reservation `status` を `refunded` にする
- refund 失敗時は必要に応じて `paid` 側へ戻す

重要:

- DB 更新に失敗したら webhook で **500** を返す
- 理由: Stripe の再送で整合を回復したいから

## customer キャンセルと返金

### 実行条件

- ログイン中 user が `customer`
- reservation の `customer_id` が本人と一致
- reservation `status` が `paid`
- `scheduled_for` が現在時刻より未来
- `stripe_payment_intent_id` がある

### 実行内容

- `POST /api/reservations/[id]/cancel`
- Stripe で全額返金を開始する
- idempotency key を使う
- refund metadata に reservation 情報を持たせる
- 返金開始直後の情報を DB に保存してよい
- 最終状態は webhook でそろえる

### 返金 UI

- customer dashboard に `refund_status` を表示する
- 状態は利用者向けの日本語へ変換する
