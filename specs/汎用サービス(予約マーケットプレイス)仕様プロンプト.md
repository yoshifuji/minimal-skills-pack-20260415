## 作るアプリ

### プロダクト名

- 画面表示名: `しっぽケア予約`
- 内容: ペットシッターがプロフィールとお世話プランを公開し、飼い主が日時を選んで予約リクエストできるサービス予約アプリ

### 画面上の呼び方

- `provider` = `ペットシッター`
- `customer` = `飼い主`
- `service_plan` = `お世話プラン`
- `reservation` = `予約`

### スコープの考え方

- このアプリは「事業者と利用者の両方がいる予約サービス」として作る
- 料金は比較と予約判断のために表示する
- 今回の対象は公開プロフィール、プラン管理、予約受付、予約状態管理までとする

## 技術スタック

必須:

- Next.js App Router
- React
- TypeScript
- Supabase Auth
- Supabase Postgres
- Supabase Storage
- `@supabase/ssr`
- Zod

方針:

- UI はプレーンな CSS を使う
- 状態管理は React 標準機能中心にする
- ORM は追加しない
- DB への読み書きは `lib/` 配下の共通処理へ寄せる
- route handler に業務ロジックをベタ書きしすぎない

## UI と文言の方針

- 日本語 UI
- 明るい背景
- 清潔感と安心感を優先する
- 過度に派手にしない
- 予約状態を利用者が理解しやすい言葉で出す
- 技術用語をそのまま出しすぎない
- 飼い主、ペットシッター、将来の運用担当が迷いにくい導線にする

## 主要ユーザー

- 飼い主
- ペットシッター
- 管理者

今回は管理者専用画面は不要です。  
ただし `admin` ロール自体は保持してください。

## 認証仕様

### サインアップ

- ルート: `/signup`
- 入力項目:
  - `fullName`
  - `email`
  - `password`
  - `role` (`customer` または `provider`)
- Supabase Auth のメールアドレスとパスワード認証を使う
- user metadata に `full_name` と `role` を保存する
- メール確認がある場合とない場合の両方に対応し、利用者が迷わない文言を出す

### ログイン

- ルート: `/login`
- email / password ログイン
- ログイン成功後の遷移:
  - `customer` → `/dashboard/customer`
  - `provider` → `/dashboard/provider`
  - `admin` → `/dashboard/provider` に仮遷移でもよいが、役割は保持する

### ログアウト

- ナビゲーションからログアウトできる

### セッション

- `@supabase/ssr` を使う
- ブラウザー用、サーバー用、管理用のクライアントを分ける
- `middleware.ts` で通常リクエスト時のセッション継続を扱う

## DB モデル

テーブル名は将来の別アプリへ流用しやすい中立名を使ってください。

### 1. `profiles`

- `id uuid primary key` references `auth.users`
- `full_name text not null`
- `role text not null default 'customer'`
- role の許可値:
  - `customer`
  - `provider`
  - `admin`
- `created_at`
- `updated_at`

### 2. `provider_profiles`

- `user_id uuid primary key` references `profiles`
- `display_name text not null default ''`
- `introduction text not null default ''`
- `service_area text not null default ''`
- `avatar_image_path text`
- `is_public boolean not null default false`
- `accepting_new_reservations boolean not null default false`
- `created_at`
- `updated_at`

### 3. `service_plans`

- `id uuid primary key default gen_random_uuid()`
- `provider_id uuid not null`
- `title text not null`
- `description text not null default ''`
- `duration_minutes integer not null`
- `price_amount integer not null`
- `currency text not null default 'jpy'`
- `cover_image_path text`
- `is_active boolean not null default true`
- `created_at`
- `updated_at`

制約:

- `duration_minutes > 0 and duration_minutes <= 480`
- `price_amount > 0`

### 4. `reservations`

- `id uuid primary key default gen_random_uuid()`
- `service_plan_id uuid not null`
- `provider_id uuid not null`
- `customer_id uuid not null`
- `scheduled_for timestamptz not null`
- `pet_name text not null default ''`
- `pet_type text not null default ''`
- `notes text not null default ''`
- `status text not null default 'pending'`
- `status_note text not null default ''`
- `price_amount integer not null`
- `currency text not null default 'jpy'`
- `cancelled_at timestamptz`
- `responded_at timestamptz`
- `completed_at timestamptz`
- `created_at`
- `updated_at`

status の許可値:

- `pending`
- `confirmed`
- `declined`
- `cancelled`
- `completed`

### 共通 DB 要件

- `updated_at` 自動更新 trigger を用意する
- `auth.users` insert 時に `profiles` を自動作成する
- role が `provider` の場合は `provider_profiles` も自動作成する
- 主要な検索列に index を付ける

最低限の index:

- `provider_profiles.is_public`
- `service_plans.provider_id`
- `service_plans.is_active`
- `reservations.provider_id`
- `reservations.customer_id`
- `reservations.scheduled_for`
- `reservations.status`

## 行レベルのアクセス制御

必須:

- `profiles`
  - 本人のみ read / update 可
- `provider_profiles`
  - 公開中 (`is_public = true`) の行は公開 read 可
  - provider 本人は own row を read / update 可
- `service_plans`
  - `is_active = true` かつ公開中 provider の plan は公開 read 可
  - provider 本人は own plans を作成、更新、停止できる
- `reservations`
  - customer 本人は own reservations を read / create 可
  - provider 本人は own incoming reservations を read / update 可

注意:

- クライアント側の表示制御だけで済ませない
- API 側でも role と所有者確認を行う

## 予約フロー

### 予約できる条件

- ログイン中 user が `customer`
- provider profile が公開中
- provider が予約受付中
- service plan が有効
- `scheduled_for` が未来日時

### 予約作成

フロー:

1. customer 認証を確認する
2. service plan を取得する
3. provider の公開状態と受付状態を確認する
4. 価格と通貨をサーバー側で確定する
5. reservation を `pending` で作成する
6. reservation 詳細画面または customer dashboard へ戻せる response を返す

保存方針:

- `price_amount` と `currency` は予約時点の値を reservation に保存する
- 後から service plan の価格が変わっても過去予約の表示は崩さない
- customer が自分以外の `customer_id` を指定できないようにする

### provider の承認 / 辞退

- `POST /api/reservations/[id]/respond`
- ログイン中 user が `provider`
- reservation の `provider_id` が本人と一致
- `pending` の予約だけを `confirmed` または `declined` に更新できる
- 応答時に `status_note` と `responded_at` を保存してよい
- すでに `cancelled` または `completed` の予約は更新しない

### customer のキャンセル

- `POST /api/reservations/[id]/cancel`
- ログイン中 user が `customer`
- reservation の `customer_id` が本人と一致
- reservation `status` が `pending` または `confirmed`
- `scheduled_for` が現在時刻より未来
- キャンセル時は `status = cancelled` と `cancelled_at` を保存する

### provider の完了処理

- `POST /api/reservations/[id]/complete`
- ログイン中 user が `provider`
- reservation の `provider_id` が本人と一致
- `confirmed` の予約だけを `completed` にできる
- 完了時は `completed_at` を保存する
